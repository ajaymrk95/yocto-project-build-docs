# Device Trees, DTBs & Configuration Files

<span class="phase-label">Phase 2 · Page 1 of 5</span>

!!! abstract "Page Goal"
    Understand the three Device Tree file formats (DTS, DTSI, DTB), how they describe hardware to the Linux kernel, how they are compiled and deployed, and which files change when adapting the Yocto build from the NVIDIA DevKit to the ConnectTech Elroy carrier board.

---

## Device Tree File Formats

Before delving into specific hardware platforms, we must first understand the files that constitute a Device Tree. There are three key file types in the Device Tree ecosystem:

### Source (`*.dts`)
The device tree structure is defined using a specialized language called Device Tree Source (DTS). The file containing structures in this format has the `.dts` extension. These files define reusable hardware components or configurations that can be included in other device tree source files using the `#include` directive. Human-readable and editable, these files act as the master layout for a specific board or platform.

### Includes (`*.dtsi`)
By using `.dtsi` (Device Tree Source Include) files, developers can organize and modularize their device tree code in a flexible and scalable way, making it easier to maintain and reuse common configurations across different device tree files. A `.dtsi` file typically describes a shared chip-level configuration (such as the Tegra 186 SoC itself) that multiple individual board-level `.dts` files can include and extend.

### Binary Blob (`*.dtb`)
The DTS file is compiled into a binary format called Device Tree Blob (DTB) that is loaded by the bootloader. The file containing this binary has the `.dtb` extension. The Linux kernel needs device tree binaries (`*.dtb`) to recongnise any external hardware/peripherals on bootup. These binaries are generated by the device tree compiler (`dtc`) from the device tree source files. The compiler is part of the Linux sources and is automatically built if needed. The kernel build system provides the `dtbs` target which compiles all device trees which are compatible with the current kernel configuration.

| File Type | Extension | Format | Purpose |
|-----------|-----------|--------|---------|
| Device Tree Source | `.dts` | Human-readable text | Top-level board/platform hardware description — editable |
| Device Tree Include | `.dtsi` | Human-readable text | Reusable hardware fragments — shared across boards via `#include` |
| Device Tree Blob | `.dtb` | Compiled binary | Kernel-readable binary — loaded by bootloader at boot time |

## Where Device Tree Files Live

On ARM architectures, all Device Tree Source files and includes are located in `arch/arm/boot/dts` on the Linux kernel source code. A tool, with source code located in `scripts/dtc`, called the Device Tree Compiler compiles the source into a binary form. The Device Tree Blob is produced by the compiler, and is the binary that gets loaded by the bootloader and parsed by the kernel at boot time. The `arch/arm/boot/dts/Makefile` lists which DTBs should be generated at build time.

!!! note "64-bit ARM Platforms (ARM64)"
    For 64-bit ARM platforms like the Jetson TX2i (which uses the Tegra 186 SoC), the device tree sources and includes are located in `arch/arm64/boot/dts/` and specifically the NVIDIA Tegra subfolder `arch/arm64/boot/dts/nvidia/`. The compilation mechanism, DTC tool path (`scripts/dtc/`), and Makefile-driven build process remain identical.

## P3489 — TX2i Development Kit Device Tree Files

The **P3489** is NVIDIA's board number for the Jetson TX2i Development Kit carrier board (used in Phase 1).

When building with `MACHINE = "jetson-tx2-devkit-tx2i"` in Yocto, the build system pulls the default NVIDIA DevKit device tree source. The primary device tree blob generated for this platform is:
- **`tegra186-quill-p3489-1000-a00-00-ucm1.dtb`** (or `tegra186-quill-p3489-1000-a00-00.dtb` depending on L4T release).

This DTB contains descriptions of the DevKit's large array of physical ports, including the HDMI connector, multi-lane PCIe slots, M.2 interfaces, debug serial ports, and standard 40-pin expansion headers. It is the baseline from which we start our adaptation.

## ConnectTech Elroy — ASG002 and Board Versions

The **Connect Tech Elroy** is identified by the part number **ASG002**. When moving from the P3489 DevKit to the Elroy carrier board, the device tree files must be changed because the physical carrier board layout is completely different.

### Finding the Specific Board Version

ConnectTech carriers have different hardware revisions and options. To find the specific Board Version, look for the hardware markings on the PCB or check the shipping manifest.
- **Revision F+** is the most general and widely deployed revision. The standard device tree binary for the TX2i on Elroy Revision F+ is **`tegra186-quill-p3489-1000-a00-00-cti-ASG002-revF+.dtb`**.
- There are also specific variants based on the expansion slots and I/O routing:
    - **MPCIE Variant**: Configured specifically to support a Mini PCIe card. Its device tree is **`tegra186-quill-p3489-1000-a00-00-cti-ASG002-mPCIE.dtb`**.
    - **USB+ Variant**: Configured for additional USB interfaces or USB 3.0 routing. Its device tree is **`tegra186-quill-p3489-1000-a00-00-cti-ASG002-USB3.dtb`**.

!!! warning "Selecting the Correct Variant"
    Using a DTB that does not match your specific Elroy board variant can lead to non-functioning ports (e.g. USB ports not power-cycling or Mini-PCIe cards not being detected) or GPIO pin collisions. Verify the board version and variant before flashing.


## CFG & BCT Files in the NVIDIA Flash Process

While Device Trees describe hardware to the kernel at **boot time**, CFG (Configuration) and BCT (Boot Configuration Table) files describe the board configuration to the bootloader and flash tool at **flash/boot time**.


### Device Tree Blobs (DTBs)

| Platform Config | Property | DTB Filename | Description |
|-----------------|----------|--------------|-------------|
| **NVIDIA Jetson TX2i DevKit** | `DTB_FILE` & `TBCDTB_FILE` | `tegra186-quill-p3489-1000-a00-00-ucm1.dtb` | Default device tree blob for the TX2i on the standard P3489 DevKit carrier. |
| **NVIDIA Jetson TX2i DevKit** | `BPFDTB_FILE` | `tegra186-a02-bpmp-storm-p3489-a00-00-ta795sa-ucm1.dtb` | BPMP (Boot and Power Management Processor) DTB configuration. |
| **CTI Elroy Rev F+** | `DTB_FILE` & `TBCDTB_FILE` | `tegra186-tx2i-cti-ASG002-revF+.dtb` | Connect Tech Elroy Revision F+ carrier board device tree blob. |
| **CTI Elroy Rev F+** | `BPFDTB_FILE` | `tegra186-a02-bpmp-storm-p3489-a00-00-ta795sa-ucm1.dtb` | Sourced from the common configuration for the BPMP firmware. |

### Boot Configuration Table (BCT) & CFG Files

The following configuration files are loaded into the Boot ROM and Early Boot stage:
- These are mainly specific to the TX2i itself and are slightly modified or left unchaged by most carrier board developers.

| Parameter | Configuration Filename | Purpose & Target |
|-----------|------------------------|------------------|
| **`EMMC_BCT`** | `P3489_A00_8GB_Samsung_8GB_lpddr4_204Mhz_P134_A02_ECC_en_l4t.cfg` | Configures early eMMC storage and DRAM timings (LPDDR4 at 204 MHz, ECC enabled) for the TX2i SoM. |
| **`PINMUX_CONFIG`** | `tegra186-mb1-bct-pinmux-quill-p3489-1000-a00.cfg` | Pin Multiplexing — Defines initial state, direction, and pull-up/pull-down values of the SoC pins. |
| **`PMIC_CONFIG`** | `tegra186-mb1-bct-pmic-quill-p3489-1000-a00.cfg` | Power Management IC — Defines output voltage rails, limits, and power-up sequence timings. |
| **`PMC_CONFIG`** | `tegra186-mb1-bct-pad-quill-p3489-1000-a00.cfg` | Pad Power Control — Sets voltage domains (e.g., 1.8V vs 3.3V) for I/O pad groups. |
| **`PROD_CONFIG`** | `tegra186-mb1-bct-prod-storm-p3489-1000-a00.cfg` | Production settings — Registers configuration for chip performance and calibration. |
| **`BOOTROM_CONFIG`** | `tegra186-mb1-bct-bootrom-quill-p3489-1000-a00.cfg` | Configures Boot ROM execution parameters. |

## The Role of Pinmux and Power Configurations

A crucial aspect of carrier board adaptation is configuring the pinmux and power control files. These configurations directly determine the **power and voltage consumption** of the system:

1. **Voltage Domain Selection (`PMC_CONFIG`)**: Different hardware peripherals require different signal voltages. The PMC configuration maps internal SoM voltage domains to match the carrier's hardware rails (e.g., configuring a pad group for 1.8 V or 3.3 V). Driving a pad at the wrong voltage can cause permanent damage to the peripheral or SoM, or result in excessive leakage current.
2. **Current Leakage Prevention (`PINMUX_CONFIG`)**: Setting pins in an incorrect electrical state (for instance, leaving unused pins "floating", or driving an active-high output against an external ground pull) can cause high leakage current. Pinmux configuration ensures all unused or non-connected pins are properly tri-stated or driven to pull-down/pull-up states, which minimizes power consumption during normal and low-power sleep cycles.
3. **Active Power Management (`PMIC_CONFIG`)**: The PMIC configuration defines voltage rails and power scaling rules. The PMIC manages the module's core voltage dynamically based on thermal limits and clock frequencies to optimize power dissipation.
